CMU 15-445 Lab3 Task1 通关，实现基本执行器和 IndexScan 优化器

📅 项目信息

  • 课程：CMU 15-445 Database Systems
  • 实验：Lab3 - Query Execution
  • 任务：Task1 - Access Method Executors
  • 耗时：2 天
  • 状态：本地全测试通过
  • 代码分支：lab3-Task1-Access-Method-Executor

前言

Lab3 需要对 Bustub 大多数源代码有基本的认识，需要静下心来阅读源代码，理解项目框架，要阅读数十个类，开始几天真是一行代码都写不出来。

核心问题

Lab1 实现缓冲池管理器，实现对磁盘的访问，Lab2 用 B+ 树实现索引结构，Lab3 需要把它们串联起来，我们需要开始对输入的 sql 语句进行响应了。

Task 1 的目标是实现数据库最底层的存储访问层执行器，让查询引擎能够真正地从存储系统读取和修改数据。

核心组件

相比之前的 Lab 有点小多，不一一列举，供上官方给出了查询处理流程图：

整体架构从上到下分为 4 个层次:

查询处理层（Query Processing）

组件名称 功能
SQL（结构化查询语言） 用户与数据库交互的语言入口
Parser（解析器） 将SQL字符串转换为抽象语法树（AST
Binder（绑定器） 将AST中的名称（如表名、列名）绑定到数据库中的实际对象
Planner（计划器） 将绑定后的查询转换为逻辑执行计划树
Optimizer（优化器） 对执行计划进行优化，选择最有效的执行方式

执行引擎层（执行计划到数据）

组件名称 功能
Executors（执行器集群） 实际执行查询计划中每个操作的组件

存储管理层（数据组织与缓存）

组件名称 功能
Index（索引） 提供快速数据查找的数据结构（Lab2 实现的B+树）
Table Heap（表堆） 存储表中所有元组的堆结构
Buffer Pool Manager（缓冲池管理器） 管理内存中的数据页缓存

磁盘管理层（持久化存储）

组件名称 功能
Disk Manager（磁盘管理器） 直接在磁盘文件上进行读写操作

数据流向：

SQL → Parser → Binder → Planner → Optimizer → Executors → Storage → Disk

核心概念

RID

  • 定义：元组在磁盘上的绝对坐标

Meta（元数据）

  • 定义：修饰关于数据本身的信息（例如：是否被删除），而不是实际的数据内容。

Tuple（元组）

  • 定义：关系数据库中的基本数据单位，对应表中一行记录。

  • 组成：

    多个字段值的有序集合

    每个字段对应表中的一个列

    存储在Page（数据页）中

  • 特性：

    有固定的结构（由Schema定义）

    通过RID（记录标识符：页号+槽号）唯一标识

    在BusTub中以二进制字节数组形式存储

Page （数据页）

  • 定义：数据库在磁盘和内存之间传输数据的基本单位。

  • 管理：

    由Buffer Pool Manager管理内存缓存

    通过Disk Manager进行磁盘读写

Schema （模式）

  • 定义：描述表结构的元数据集合。

  • 包含信息：

    列的数量和顺序

    每列的：名称、数据类型、长度、是否可为 NULL

Table （表）

  • 定义：关系数据库中的主要数据组织形式，是逻辑数据结构。

  • 关键误区： 表是逻辑数据结构，在磁盘上并没有一块区域的页面明确表示一个表，表是由表的元数据（存放在系统目录）和数据页构成的集合。

  • 物理实现：

  1. 系统目录中存储元数据
  2. Table Heap管理数据页

Index （索引）

  • 定义：数据的”快速查找目录” ，Key → 数据位置（RID）

  • 物理实现：所有相同相同索引列组成的索引都会存放在同一棵 B+ 树内。

System Catalog （系统目录）

  • 定义：数据库的元数据管理中心，也称为数据字典。

  • 存储内容：
    表信息，索引信息，表-索引映射关系

执行器火山模型

执行器火山模型是数据库查询执行的核心架构，又称迭代器模型。它采用统一的迭代器接口，将查询计划组织成树形结构，数据像火山喷发一样从底层向上层流动。
